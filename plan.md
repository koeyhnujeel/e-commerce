# 개발 계획: 상품 조회, 주문, 결제 MVP

## 1. 목표

`spec.md`의 범위를 한 번에 크게 구현하지 않고, 단순한 조회 기능부터 시작해서 주문과 결제를 순차적으로 쌓아 올리는 방식으로 개발한다.

핵심 원칙은 아래와 같다.

- 먼저 읽기 기능을 완성해 도메인 구조를 안정화한다.
- 그다음 주문 생성/조회로 구매 플로우의 중심을 만든다.
- 수정/삭제와 상태 전이를 붙여 주문 도메인을 완성한다.
- 마지막에 토스페이먼츠 테스트베드 연동으로 결제 흐름을 붙인다.

## 2. 전제와 기술 방향

### 아키텍처

- 단일 Spring Boot 애플리케이션
- Spring Modulith 기반 모듈 분리
- 모듈 후보
  - `catalog`
  - `order`
  - `payment`
  - `like`
  - `member`

### 백엔드 기술

- Kotlin + Spring Boot 3.5
- Spring Web
- Spring Data JPA
- MySQL
- Spring Security + JWT
- Bean Validation
- MockK, JUnit5, Kotest assertions

### 결제 기술

- PG: 토스페이먼츠 테스트베드
- 프런트 결제 UI: 토스페이먼츠 JS SDK v2
- 서버 연동: 토스 결제 승인 API, 결제 취소 API
- HTTP 클라이언트: Spring `RestClient` 권장

### 연동 원칙

- 프런트는 토스 결제창 또는 결제위젯을 호출한다.
- 서버는 주문/결제 상태를 소유한다.
- 토스 리다이렉트 성공 후에도 서버 승인 API 호출 전까지 결제 완료로 간주하지 않는다.
- 서버는 토스가 전달한 `paymentKey`, `orderId`, `amount`를 내부 주문 데이터와 검증한 뒤 승인한다.

### 환경변수 초안

- `TOSS_WIDGET_CLIENT_KEY`
- `TOSS_WIDGET_SECRET_KEY`
- `TOSS_SUCCESS_URL`
- `TOSS_FAIL_URL`
- `TOSS_API_BASE_URL`

테스트베드 기본값은 별도 프로파일 또는 환경변수로 관리한다.

## 3. 공통 선행 작업

모든 phase에 앞서 아래 정리를 먼저 수행한다.

- `spec.md` 기준 용어 확정
- `order` 상태 모델 재정의
- `payment` 모듈 패키지 자리 확보
- 필요한 `ErrorCode` 추가 초안 정의
- `SecurityConfiguration`에 신규 인증 엔드포인트 반영 계획 수립

권장 공통 개선사항은 아래와 같다.

- `Product`에 판매 상태 필드 추가
- `Order`에 `orderNumber`, `totalAmount` 추가
- `OrderItem`에 상품/옵션 스냅샷 필드 추가
- `Payment`용 공통 시간 필드와 상태 enum 설계

## 4. Phase 계획

## Phase 1. 상품 조회 기반 만들기

### 목표

사용자가 상품 목록과 상세를 조회할 수 있게 만들고, 이후 주문 생성이 가능한 상품 응답 구조를 먼저 확정한다.

### 범위

- 상품 목록 조회 API
- 상품 상세 조회 API
- 대표 이미지/옵션/판매 상태 응답 구성
- 비회원 조회 지원

### 구현 작업

- `catalog` 조회용 application service 추가
- 목록/상세 응답 DTO 추가
- 페이지네이션과 정렬 조건 추가
- 상품 공개 상태 필드 또는 메서드 추가
- `ProductJpaRepository` 기반 조회 쿼리 추가

### 기술 포인트

- 목록 조회는 `Pageable` 기반으로 시작한다.
- 상세 조회는 JPA fetch 전략을 제어해 N+1을 피한다.
- 대표 이미지는 도메인 메서드 또는 전용 mapper에서 선택한다.
- 리뷰는 응답에 포함하지 않는다.

### 완료 기준

- `GET /api/products`
- `GET /api/products/{productId}`
- 비로그인 상태에서 정상 조회 가능
- 상품 목록/상세 통합 테스트 통과

## Phase 2. 좋아요 읽기 연동

### 목표

상품 조회 응답에 좋아요 수와 내 좋아요 여부를 포함시켜 실제 사용자 관점의 상품 조회를 완성한다.

### 범위

- 상품별 좋아요 수 조회
- 회원 기준 좋아요 여부 조회
- 목록 응답용 벌크 좋아요 집계

### 구현 작업

- `like` 모듈에 조회 포트/API 추가
- 상품 ID 목록 기준 집계 쿼리 추가
- `catalog` 응답 조합 로직 추가
- 로그인/비로그인 분기 처리

### 기술 포인트

- 목록 조회에서는 상품별 개별 쿼리가 아니라 벌크 조회를 사용한다.
- `CurrentMember`가 없을 때 `likedByMe = false`로 처리한다.
- `catalog`가 `like`에 직접 영속성 접근하지 않고 공개 API로만 조회한다.

### 완료 기준

- 상품 목록에 좋아요 수 포함
- 상품 상세에 좋아요 수와 `likedByMe` 포함
- 로그인/비로그인 케이스 테스트 통과

## Phase 3. 주문 생성, 목록, 상세

### 목표

결제 전 단계의 주문 생성과 조회를 먼저 구현해서 구매 플로우의 중심 엔티티를 안정화한다.

### 범위

- 주문 생성
- 내 주문 목록 조회
- 내 주문 상세 조회

### 구현 작업

- `Order` 상태와 생성 규칙 재설계
- `OrderItem` 스냅샷 필드 추가
- 주문 총액 계산 로직 추가
- `POST /api/orders`
- `GET /api/orders`
- `GET /api/orders/{orderId}`

### 기술 포인트

- 주문 생성 시 `catalog` 공개 API를 통해 상품/옵션 존재 여부를 검증한다.
- 주문 금액은 클라이언트 입력값을 신뢰하지 않고 서버가 계산한다.
- 토스 `orderId`에 대응할 수 있도록 내부 `orderNumber`는 문자열 기반 유니크 값으로 설계한다.
- 주문 상세 응답은 이후 결제 화면에 그대로 재사용 가능하도록 구성한다.

### 완료 기준

- 인증 회원만 주문 생성/조회 가능
- 타인 주문 접근 차단
- 주문 총액 계산 및 스냅샷 저장 검증
- 주문 API 통합 테스트 통과

## Phase 4. 주문 수정, 삭제, 상태 전이 정리

### 목표

결제 연결 전에 주문 도메인의 변경 가능 구간과 불변 조건을 명확히 고정한다.

### 범위

- 주문 수정
- 주문 삭제 또는 소프트 삭제
- 결제 전/후 상태 전이 규칙 확정

### 구현 작업

- `PATCH /api/orders/{orderId}`
- `DELETE /api/orders/{orderId}`
- `OrderStatus`와 `OrderItemStatus` 정리
- 주문 변경 가능 여부 판단 메서드 추가
- 결제 전용 상태 `PAYMENT_PENDING` 반영

### 기술 포인트

- 주문 수정은 전체 주문상품 재구성 방식으로 단순하게 시작한다.
- 삭제는 물리 삭제보다 소프트 삭제를 우선 검토한다.
- 향후 배송/환불 확장을 고려해 상태 전이 메서드를 엔티티에 둔다.

### 완료 기준

- 결제 전 주문만 수정/삭제 가능
- 결제 완료 주문 수정/삭제 차단
- 주문 상태 전이 도메인 테스트 통과

## Phase 5. 결제 모듈과 토스 체크아웃 준비

### 목표

실제 승인 호출 전까지의 결제 준비 단계를 만들고, 토스 UI에서 사용할 체크아웃 데이터 계약을 확정한다.

### 범위

- `payment` 모듈 생성
- 결제 요청 생성 API
- 토스 체크아웃용 응답 설계
- 최소 수동 테스트 페이지 또는 샘플 연동 지원

### 구현 작업

- `Payment` 엔티티, 상태 enum, repository 추가
- `POST /api/orders/{orderId}/payments`
- 주문과 결제 1:1 활성 관계 제약 추가
- 결제 요청 생성 시 주문 상태를 `PAYMENT_PENDING`으로 변경
- 토스 요청용 데이터 응답
  - `orderId`
  - `orderName`
  - `amount`
  - `customerKey`
  - `successUrl`
  - `failUrl`

### 기술 포인트

- `customerKey`는 회원 식별값 기반의 외부 노출용 안정 키로 만든다.
- 프런트가 아직 없으면 `src/main/resources/static` 아래 테스트용 체크아웃 HTML을 두는 방식을 허용한다.
- 토스 JS SDK는 v2를 사용한다.
- 결제 요청 단계에서는 아직 승인하지 않는다.

### 완료 기준

- 주문 기준 결제 요청 생성 가능
- 중복 활성 결제 생성 차단
- 토스 체크아웃 호출에 필요한 데이터 제공

## Phase 6. 토스 승인, 취소, 실패 처리

### 목표

토스 테스트베드 기준으로 실제 결제 승인과 취소를 연결하고, 내부 주문/결제 상태를 일관되게 맞춘다.

### 범위

- 결제 확정 API
- 결제 취소 API
- 실패 처리
- 토스 응답 저장

### 구현 작업

- 토스 API 클라이언트 추가
- `POST /api/payments/{paymentId}/confirm`
- `POST /api/payments/{paymentId}/cancel`
- 토스 승인 응답의 `paymentKey`, 결제수단, 카드 정보 일부 저장
- 실패 사유 저장
- 주문 상태 `PAID`, `CANCELED` 연동

### 기술 포인트

- 서버는 토스 성공 리다이렉트로 전달된 `orderId`, `amount`를 내부 데이터와 비교한다.
- 서버는 승인 시 토스 secret key 기반 Basic 인증을 사용한다.
- 승인 성공 전에는 `APPROVED` 상태로 바꾸지 않는다.
- 취소는 내부 결제 상태만 바꾸지 말고 토스 취소 API 결과와 함께 반영한다.
- 테스트베드에서는 실제 과금이 발생하지 않는 키로 검증한다.

### 완료 기준

- 토스 테스트베드로 결제 승인 가능
- 승인 후 주문이 `PAID`로 전이
- 취소 후 주문/결제 상태가 정책대로 변경
- 승인 실패/재시도 정책 테스트 통과

## Phase 7. 안정화와 운영 준비

### 목표

MVP 배포 전 정합성, 테스트, 운영 설정을 보강한다.

### 범위

- 예외 코드 정리
- 로그/모니터링 포인트 정리
- 환경변수와 설정 문서 정리
- Modulith 경계 검증

### 구현 작업

- `ErrorCode` 정리
- 외부 API 호출 로그 마스킹
- 결제/주문 상태 변경 이벤트 로그 추가
- `ModulithArchitectureTests` 업데이트
- 로컬/테스트베드 실행 가이드 정리

### 기술 포인트

- 민감정보인 secret key, 카드 세부정보는 로그에 남기지 않는다.
- 외부 API 오류와 도메인 오류를 분리해 다룬다.
- 추후 webhook을 붙일 수 있게 `paymentKey`와 외부 참조값 저장 구조를 유지한다.

### 완료 기준

- 주요 시나리오 테스트 통과
- 모듈 경계 테스트 통과
- 환경설정 문서 정리 완료

## 5. 권장 구현 순서 요약

1. Phase 1로 상품 조회를 먼저 안정화한다.
2. Phase 2로 좋아요 읽기를 붙인다.
3. Phase 3으로 주문 생성/조회의 핵심 흐름을 만든다.
4. Phase 4로 주문 수정/삭제와 상태 규칙을 고정한다.
5. Phase 5로 `payment` 모듈과 체크아웃 준비를 붙인다.
6. Phase 6으로 토스 테스트베드 승인/취소를 연결한다.
7. Phase 7로 테스트, 설정, 운영 정리를 마무리한다.

## 7. 기술 리스크와 대응

### 리스크 1. 현재 저장소가 백엔드 중심이다

- 대응: 토스 결제 UI는 별도 프런트가 호출하도록 API 계약을 먼저 만든다.
- 대응: 필요하면 테스트용 정적 HTML 페이지를 임시 제공한다.

### 리스크 2. 기존 주문 도메인이 구매확정 중심이다

- 대응: Phase 3~4에서 주문 상태 모델을 먼저 재설계한다.
- 대응: 리뷰와 연결된 기존 구매확정 흐름은 깨지지 않게 호환 계층을 둔다.

### 리스크 3. 결제 승인 타이밍 불일치

- 대응: 서버 승인 API 호출 전에는 결제 완료 처리하지 않는다.
- 대응: `orderId`, `amount` 서버 검증을 강제한다.

### 리스크 4. 모듈 경계 침범

- 대응: 직접 repository 접근 대신 공개 API/port를 우선 사용한다.
- 대응: 마지막 phase에서 Modulith 테스트로 확인한다.
