# SPEC: 상품 조회, 주문, 결제 MVP

## 1. 문서 목적

현재 저장소의 구현 상태를 기준으로 아래 기능을 설계하고 구현 범위를 명확히 정의한다.

- 상품 목록 조회
- 상품 상세 조회
- 주문 CRUD
- 결제

이 문서는 Spring Modulith 구조를 유지하면서 `catalog`, `order`, `payment`, `member`, `like` 모듈 간 책임과 API 계약을 정리하는 것을 목표로 한다.

## 2. 현재 상태

### 이미 구현됨

- 회원가입/인증 및 JWT 기반 인증
- 상품 좋아요 등록/취소
- 리뷰 도메인 및 일부 기능
- 주문의 일부 도메인과 구매 확정 기능
- 상품 엔티티 및 옵션/이미지 엔티티

### 현재 한계

- 상품은 엔티티만 존재하고 목록/상세 조회 API가 없다.
- 주문은 구매 확정 외의 생성/조회/수정/삭제 기능이 없다.
- 결제 모듈이 없다.
- 좋아요는 쓰기 API만 있고, 상품 응답에 필요한 조회 API가 없다.
- 리뷰는 미완성이지만 이번 범위에서 고려하지 않는다.

## 3. 이번 작업의 원칙

- 회원과 좋아요는 기존 기능을 존중하면서 연동한다.
- 리뷰는 이번 작업에서 완전히 제외한다.
- 상품/주문 도메인 엔티티는 요구사항에 맞게 수정 가능하다.
- 결제는 토스페이먼츠 테스트베드 기준으로 구현한다.
- 모든 공개 API는 기존 `ApiResponse` 형식을 따른다.

## 4. 목표

### 제품 목표

- 사용자가 상품 목록과 상세를 조회할 수 있다.
- 로그인 사용자가 좋아요 상태를 포함한 상품 정보를 조회할 수 있다.
- 로그인 사용자가 주문을 생성, 조회, 수정, 삭제할 수 있다.
- 로그인 사용자가 주문에 대해 결제를 생성하고 승인할 수 있다.

### 구현 목표

- `catalog` 모듈에 조회용 애플리케이션 서비스와 API를 추가한다.
- `order` 모듈을 주문 생명주기 중심으로 재구성한다.
- `payment` 모듈을 신규 추가한다.
- 모듈 간 조회/검증용 공개 API를 최소 범위로 정의한다.

## 5. 범위

### 포함

- 상품 목록 조회
- 상품 상세 조회
- 상품 응답 내 좋아요 수/내 좋아요 여부 제공
- 주문 생성
- 내 주문 목록 조회
- 내 주문 상세 조회
- 미결제 주문 수정
- 미결제 주문 삭제
- 주문 결제 요청
- 결제 승인
- 결제 실패/취소 상태 반영
- 토스페이먼츠 테스트베드 연동

### 제외

- 리뷰 작성/조회/노출
- 상품 등록/수정/삭제용 관리자 기능
- 장바구니

- 배송 추적

- 토스페이먼츠 라이브 키 전환
- 환불 정산

## 6. 핵심 사용자 시나리오

### 상품 조회

1. 비로그인 사용자는 상품 목록을 조회할 수 있다.
2. 로그인 사용자는 상품 목록에서 각 상품의 `likedByMe`를 확인할 수 있다.
3. 사용자는 상품 상세에서 이미지, 옵션, 가격, 좋아요 정보, 판매 가능 상태를 확인할 수 있다.

### 주문

1. 로그인 사용자는 상품 옵션과 수량을 선택해 주문을 생성할 수 있다.
2. 사용자는 자신의 주문 목록과 상세를 조회할 수 있다.
3. 사용자는 결제 전까지만 주문 정보를 수정할 수 있다.
4. 사용자는 결제 전까지만 주문을 삭제할 수 있다.

### 결제

1. 로그인 사용자는 생성된 주문에 대해 결제를 요청할 수 있다.
2. 프런트엔드는 토스페이먼츠 결제 UI를 통해 결제를 진행할 수 있다.
3. 결제 승인 시 주문 상태가 결제 완료 상태로 변경된다.
4. 토스 결제 실패 시 주문은 다시 결제 가능한 상태를 유지한다.
5. 주문 취소 또는 결제 취소 시 결제 상태와 주문 상태가 일관되게 변경된다.

## 7. 상세 요구사항

### 7.1 상품 목록 조회

#### 기능 요구사항

- `GET /api/products`
- 페이지네이션을 지원한다.
- 카테고리 ID 필터를 지원한다.
- 정렬 기준을 지원한다.
  - `LATEST`
  - `PRICE_ASC`
  - `PRICE_DESC`
- 판매 가능한 상품만 노출한다.
- 각 상품은 아래 정보를 포함한다.
  - 상품 ID
  - 상품명
  - 대표 이미지 URL
  - 기본 가격
  - 좋아요 수
  - 로그인 사용자의 좋아요 여부
  - 판매 상태

#### 응답 정책

- 비로그인 사용자의 `likedByMe`는 `false`로 응답한다.
- 대표 이미지는 `isPrimary = true` 이미지를 사용한다.
- 옵션 전체를 목록 응답에 모두 싣지 않고 최소 정보만 제공한다.

### 7.2 상품 상세 조회

#### 기능 요구사항

- `GET /api/products/{productId}`
- 판매 가능한 상품만 조회 가능하다.
- 상세 응답은 아래 정보를 포함한다.
  - 상품 ID
  - 상품명
  - 설명
  - 기본 가격
  - 카테고리 ID 목록
  - 이미지 목록
  - 옵션 목록
  - 좋아요 수
  - 로그인 사용자의 좋아요 여부
  - 판매 상태

#### 옵션 응답

- 옵션 ID
- 색상
- 사이즈
- 재고 식별자
- 주문 가능 여부

#### 비고

- 리뷰 정보는 포함하지 않는다.
- 좋아요 수와 `likedByMe`는 `like` 모듈을 통해 조회한다.

### 7.3 주문 CRUD

#### 생성

- `POST /api/orders`
- 로그인 사용자의 주문만 생성할 수 있다.
- 요청은 하나 이상의 주문 상품을 포함해야 한다.
- 각 주문 상품은 `productId`, `productOptionId`, `quantity`를 포함한다.
- 상품/옵션 존재 여부를 검증한다.
- 주문 시점의 상품명, 옵션명, 가격은 스냅샷으로 저장할 수 있어야 한다.

#### 조회

- `GET /api/orders`
- `GET /api/orders/{orderId}`
- 본인 주문만 조회 가능하다.
- 목록은 최신 주문 순으로 정렬한다.
- 목록 응답은 주문 요약 정보 중심으로 제공한다.
- 상세 응답은 주문 상품 목록, 금액, 결제 상태를 포함한다.

#### 수정

- `PATCH /api/orders/{orderId}`
- 결제 전 상태의 주문만 수정 가능하다.
- 주문 상품 구성 변경 또는 수량 변경을 지원한다.
- 수정 시 전체 주문 금액이 다시 계산되어야 한다.

#### 삭제

- `DELETE /api/orders/{orderId}`
- 결제 전 상태의 주문만 삭제 가능하다.
- 물리 삭제보다 소프트 삭제를 우선 검토한다.
- 이미 결제되었거나 배송 프로세스가 시작된 주문은 삭제할 수 없다.

#### 주문 상태

MVP 기준으로 아래 상태를 사용한다.

- `CREATED`: 주문 생성 완료, 아직 결제 전
- `PAYMENT_PENDING`: 결제 진행 중
- `PAID`: 결제 승인 완료
- `CANCELED`: 주문 취소
- `DELIVERED`: 배송 완료

기존 상태 체계와 충돌하면 주문/주문상품 상태 enum은 이번 작업에서 조정 가능하다.

### 7.4 결제

#### 범위 정의

- 토스페이먼츠 테스트베드 연동을 기준으로 구현한다.
- 프런트엔드는 토스페이먼츠 JS SDK v2를 사용해 결제창 또는 결제위젯을 호출한다.
- 백엔드는 주문/결제 도메인 관리와 토스 승인 API 연동을 담당한다.
- 결제 수단은 MVP에서 `CARD` 단일 타입으로 시작한다.
- 결제는 한 주문당 하나의 활성 결제만 허용한다.

#### 기능 요구사항

- `POST /api/orders/{orderId}/payments`
  - 주문 기준 결제 요청 생성
  - 토스 결제 요청에 필요한 `orderId`, `orderName`, `amount`, `customerKey` 성격의 데이터를 응답
  - 주문 상태를 `PAYMENT_PENDING`으로 전이
- `POST /api/payments/{paymentId}/confirm`
  - 토스 성공 리다이렉트 이후 전달받은 `paymentKey`, `orderId`, `amount`를 검증
  - 토스 결제 승인 API를 호출해 최종 승인
  - 결제 상태를 `APPROVED`로 변경
  - 주문 상태를 `PAID`로 변경
- `POST /api/payments/{paymentId}/cancel`
  - 토스 결제 취소 API를 호출한다.
  - 결제 상태를 `CANCELED`로 변경
  - 주문 상태를 `CANCELED` 또는 `CREATED`로 변경할지 정책 결정 필요
- `GET /api/payments/{paymentId}`
  - 결제 상세 조회

#### 결제 상태

- `READY`
- `PENDING`
- `APPROVED`
- `FAILED`
- `CANCELED`

#### 결제 실패 정책

- 토스 승인 실패 시 결제는 `FAILED` 상태가 된다.
- 주문은 다시 결제 요청 가능한 상태로 복귀해야 한다.
- 실패 사유는 저장한다.

#### 토스페이먼츠 연동 정책

- 결제 성공 후 토스는 `successUrl`로 리다이렉트하며 `paymentKey`, `orderId`, `amount`를 전달한다.
- 백엔드는 성공 리다이렉트로 전달된 `orderId`, `amount`를 내부 주문 데이터와 비교 검증해야 한다.
- 최종 과금은 백엔드가 토스 결제 승인 API를 호출한 뒤에만 완료된 것으로 본다.
- 테스트 키는 실제 과금되지 않는 테스트베드로 사용한다.
- 토스 연동 키는 환경변수로 분리한다.

## 8. 회원/좋아요 연동 요구사항

### 회원

- 상품 조회는 비회원도 가능하다.
- 주문/결제는 인증된 회원만 가능하다.
- 주문/결제 조회는 항상 `CurrentMember` 기준으로 본인 데이터만 허용한다.

### 좋아요

- 상품 목록/상세 응답에 좋아요 수를 포함한다.
- 로그인 사용자는 `likedByMe`를 확인할 수 있어야 한다.
- 이를 위해 `like` 모듈에 아래 성격의 조회 API가 필요하다.
  - 상품별 좋아요 수 조회
  - 특정 회원의 상품 좋아요 여부 조회
  - 상품 ID 목록 기준 좋아요 집계 조회

## 9. 도메인 설계 방향

### 9.1 catalog

#### 필요 변경

- 조회 전용 서비스 추가
- 상품 목록/상세 DTO 추가
- 상품 공개 상태를 위한 필드 또는 enum 검토
- 대표 이미지/옵션 가공 로직 추가

#### 권장 도메인 보강

- `ProductStatus` 또는 `SaleStatus`
- 생성일시/수정일시
- 주문 가능 여부 판단 메서드

### 9.2 order

#### 필요 변경

- 주문 생성/조회/수정/삭제 유스케이스 추가
- 주문 총액 계산 책임 부여
- 주문 상품 스냅샷 필드 추가 검토
  - 상품명
  - 옵션 색상
  - 옵션 사이즈
  - 주문 당시 가격
- 결제 상태와 정합성 있는 상태 전이 메서드 추가
- 토스 연동용 외부 주문 식별자와 결제 가능 여부 판단 메서드 추가

#### 권장 엔티티 확장

- `orderNumber`
- `totalAmount`
- `deletedAt` 또는 `isDeleted`
- 수취인/배송지 필드

## 10. 모듈 의존 방향

- `catalog`는 상품 조회를 책임진다.
- `order`는 `catalog`의 상품/옵션 검증 API를 사용한다.
- `payment`는 `order`의 주문 조회/상태 변경 API를 사용한다.
- `catalog`는 상품 응답 강화를 위해 `like` 조회 API를 사용할 수 있다.
- `review`는 이번 범위에서 신규 의존을 추가하지 않는다.
- `payment`는 토스페이먼츠 외부 API 클라이언트를 가진다.

## 12. 비즈니스 규칙

- 상품 목록/상세는 숨김 또는 판매 중지 상품을 노출하지 않는다.
- 주문은 최소 1개 이상의 주문상품을 가져야 한다.
- 주문 상품의 수량은 1 이상이어야 한다.
- 다른 회원의 주문/결제는 접근할 수 없다.
- 결제 승인된 주문은 수정/삭제할 수 없다.
- 동일 주문에 중복 승인된 결제가 생기면 안 된다.
- 주문 총액과 결제 금액은 항상 일치해야 한다.
- 좋아요 정보는 리뷰와 독립적으로 동작해야 한다.
- 토스 리다이렉트에서 받은 `amount`는 서버 저장 금액과 반드시 일치해야 한다.
- 토스 `paymentKey`는 승인 이후 결제 식별자로 저장해야 한다.

- 타인 주문 접근
- 결제 완료 주문 수정/삭제 시도
- 이미 승인된 결제 재승인 시도
- 취소된 주문에 대한 결제 요청
- 토스 리다이렉트 파라미터 위변조

필요한 `ErrorCode`는 이번 작업에서 추가한다.

## 14. 테스트 요구사항

### domain

- 상품 조회용 상태 판단 로직 테스트
- 주문 생성/수정/삭제/상태 전이 테스트
- 결제 승인/실패/취소 테스트

### application

- `mockk` 기반으로 포트 mocking
- 주문 생성 시 상품 검증/가격 계산 흐름 검증
- 토스 승인 성공/실패 시 주문 상태 변경 연계 검증
- 좋아요 응답 조합 로직 검증

### adapter/out

- 주문/결제 영속성 매핑 검증
- 상품 조회 쿼리와 페이징 검증
- 좋아요 집계 조회 쿼리 검증
- 토스 API 클라이언트 직렬화/역직렬화 검증

### adapter/in

- 상품 목록/상세 API 통합 테스트
- 주문 CRUD API 통합 테스트
- 결제 생성/확정/취소 API 통합 테스트

### architecture

- 신규 `payment` 모듈 추가 또는 공개 API 변경 시 `ModulithArchitectureTests`를 함께 검증한다.

## 15. 구현 순서 제안

1. `catalog` 조회 API와 좋아요 조회 API를 먼저 추가한다.
2. 주문 도메인 상태와 스냅샷 구조를 정리한다.
3. 주문 생성/조회/수정/삭제 API를 구현한다.
4. `payment` 모듈과 토스 API 클라이언트를 추가한다.
5. 결제 생성/확정/취소 흐름을 구현한다.
6. 통합 테스트와 Modulith 테스트를 보강한다.

## 16. 완료 기준

- 상품 목록/상세 API가 정상 동작한다.
- 로그인 사용자 기준 좋아요 정보가 상품 응답에 포함된다.
- 주문 CRUD가 본인 주문 기준으로 동작한다.
- 토스 테스트베드 기준 결제 생성/확정/취소 흐름이 주문 상태와 일관되게 동작한다.
- 리뷰 관련 기능을 건드리지 않고도 전체 흐름이 동작한다.
- 관련 단위 테스트, 통합 테스트, Modulith 테스트가 추가된다.
